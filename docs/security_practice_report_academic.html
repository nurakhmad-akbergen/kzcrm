Отчет по обеспечению безопасности веб-приложения KZCRMS

Проект: KZCRMS
Формат документа: академическое описание выполненной практической работы по анализу и усилению безопасности веб-приложения.

Актуальность темы

В современных условиях цифровизации малого и среднего бизнеса веб-приложения класса CRM хранят и обрабатывают чувствительные данные о клиентах, сотрудниках, услугах, платежах и бизнес-процессах. Даже сравнительно небольшой SaaS-проект должен учитывать вопросы безопасной аутентификации, защиты от несанкционированных действий, устойчивости к перебору учетных данных и корректной production-конфигурации. Поэтому проверка и усиление безопасности CRM-системы является актуальной и практически значимой задачей.

Цель работы

Целью работы являлся анализ текущего состояния безопасности проекта KZCRMS, выявление наиболее значимых рисков и реализация практических мер защиты, повышающих надежность веб-приложения перед его использованием реальными клиентами.

Задачи работы

  • проанализировать настройки безопасности Django-приложения;
  • выявить уязвимые места в пользовательских сценариях и чувствительных маршрутах;
  • исключить опасные destructive-операции, выполняемые небезопасным способом;
  • добавить базовую защиту от brute force и abuse на формах аутентификации;
  • устранить утечки внутренних технических ошибок во внешний интерфейс;
  • усилить production-конфигурацию проекта;
  • подтвердить корректность изменений автоматизированными тестами.

Краткая характеристика проекта

KZCRMS представляет собой веб-приложение для сервисного бизнеса, разработанное на основе Django 3.2. Система предназначена для управления записями, клиентской базой, специалистами, услугами, оплатой и внутренней аналитикой. Приложение использует web-аутентификацию, email-подтверждение аккаунта, восстановление пароля, интеграцию входа через Google, а также разворачивается в облачной среде Railway.

Выявленные проблемы безопасности

1. Удаление данных через GET-запросы

В ходе анализа было установлено, что удаление некоторых сущностей выполнялось по обычным ссылкам. Такая реализация является небезопасной, поскольку создает риск CSRF-атаки на разрушение данных. Авторизованный пользователь мог быть вынужден выполнить нежелательное действие простым переходом по вредоносной ссылке.

2. Возможность запуска production с тестовым SECRET_KEY

Было выявлено, что при определенных условиях production-среда могла использовать fallback-значение SECRET_KEY, предназначенное только для разработки. Это противоречит требованиям безопасной эксплуатации Django-приложения.

3. Утечка внутренних ошибок пользователю

При сбое отправки письма подтверждения регистрации система выводила пользователю техническое исключение. Данный подход опасен тем, что может раскрывать внутренние особенности почтовой инфраструктуры, API-интеграций и сетевой конфигурации.

4. Отсутствие ограничения частоты попыток

Формы логина, регистрации и восстановления пароля изначально не имели встроенной защиты от большого количества запросов. Это увеличивало риск перебора паролей, спама почтовых уведомлений и злоупотребления публичными auth-маршрутами.

5. Недостаточно жесткие production-настройки

Несмотря на наличие части базовых параметров безопасности, приложение не использовало весь необходимый набор защитных настроек для production-эксплуатации, включая HSTS и ряд безопасных HTTP-политик.

Реализованные меры защиты

1. Перевод опасных операций удаления на POST

В целях устранения риска CSRF destructive-операции были переведены на POST-запросы. Дополнительно для этих операций применена встроенная защита Django через CSRF-токены. 

@login_required
@require_POST
def delete_barber(request, barber_id):
    ...

@login_required
@require_POST
def delete_service(request, service_id):
    ...

@login_required
@require_POST
def delete_payment_method(request, method_id):
    ...

На уровне шаблонов ссылки удаления были заменены на HTML-формы с методом POST.

2. Защита production-конфигурации SECRET_KEY

В конфигурации приложения реализована явная проверка, запрещающая запуск production-приложения с небезопасным тестовым ключом. 

SECRET_KEY = os.getenv("SECRET_KEY", "django-insecure-dev-only-key")

if not DEBUG and SECRET_KEY == "django-insecure-dev-only-key":
    raise ImproperlyConfigured("SECRET_KEY must be configured in production.")

3. Исключение утечки технических подробностей во внешний интерфейс

Ошибки отправки email продолжают логироваться на стороне приложения для разработчика, однако конечному пользователю показывается только безопасное и нейтральное сообщение. Таким образом, сохраняется диагностическая полезность логов без раскрытия внутренней инфраструктуры. 

except Exception as exc:
    logger.exception("Failed to send activation email during registration")
    form.add_error(
        "email",
        "Не удалось отправить письмо подтверждения. Проверь настройки почты и попробуй ещё раз."
    )

4. Добавление базового rate limiting

Для защиты от brute force была реализована базовая схема ограничения попыток по IP-адресу. Она применяется к логину, регистрации и восстановлению пароля. После превышения лимита пользователь временно блокируется и получает предупреждение. 

AUTH_RATE_LIMIT_WINDOW = 300
AUTH_RATE_LIMITS = {
    "login": 5,
    "register": 5,
    "password_reset": 5,
}

На основе этой логики были созданы специальные классы защищенных auth-view.

5. Усиление production-настроек Django

Для production-режима были включены дополнительные настройки:

  • SESSION_COOKIE_SECURE
  • SESSION_COOKIE_HTTPONLY
  • CSRF_COOKIE_SECURE
  • CSRF_COOKIE_HTTPONLY
  • SECURE_SSL_REDIRECT
  • SECURE_HSTS_SECONDS
  • SECURE_HSTS_INCLUDE_SUBDOMAINS
  • SECURE_HSTS_PRELOAD
  • SECURE_CONTENT_TYPE_NOSNIFF
  • X_FRAME_OPTIONS = "DENY"
  • REFERRER_POLICY = "same-origin"

Проверка результатов

После внесения изменений были расширены автоматические тесты, охватывающие как бизнес-логику, так и security-сценарии. Проверка включала тесты на:

  • неразглашение внутреннего текста ошибок при сбое отправки email;
  • блокировку чрезмерных попыток входа;
  • блокировку злоупотребления формой восстановления пароля;
  • запрет удаления сущностей через GET;
  • сохранение корректной работы CRM после внедрения защитных механизмов.

По результатам проверки:

  • команда manage.py check завершилась без ошибок;
  • команда manage.py test core завершилась успешно;
  • все автоматические тесты были пройдены.

Практическая значимость работы

Выполненная работа имеет практическую значимость, поскольку напрямую повышает надежность и устойчивость CRM-системы перед использованием реальными клиентами. Реализованные меры уменьшают риск несанкционированного удаления данных, усложняют сценарии перебора паролей, предотвращают утечку внутренних технических деталей и формируют более безопасную production-конфигурацию веб-приложения.

Вывод

В ходе практической работы был выполнен аудит безопасности проекта KZCRMS и реализован комплекс прикладных мер защиты. Были исправлены наиболее критичные проблемы, связанные с dangerous-операциями, конфиденциальностью системных ошибок, безопасностью аутентификации и production-конфигурацией Django. Полученный результат показывает, что даже для небольшого SaaS-продукта необходимо заранее выстраивать корректную архитектуру безопасности, а также сопровождать изменения автоматизированными тестами.
