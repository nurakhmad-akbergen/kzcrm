Отчет по безопасности проекта KZCRMS

Проект: KZCRMS
Технологический стек: Django 3.2, Tailwind CSS, Railway, Resend, Google OAuth
Назначение документа: описание выполненного аудита безопасности и реализованных защитных механизмов для отчета по практике.

1. Цель работы

Целью работы являлась проверка веб-приложения KZCRMS на наличие типовых проблем безопасности и последующее устранение наиболее критичных рисков до вывода продукта на этап работы с реальными клиентами.

2. Выявленные проблемы безопасности

2.1. Удаление данных через GET-запросы

В системе операции удаления сотрудников, услуг и способов оплаты выполнялись через обычные ссылки. Это создавало риск CSRF-атаки на разрушение данных: авторизованного пользователя можно было заставить перейти по вредоносной ссылке и удалить сущность без явного подтверждения на серверной стороне.

2.2. Возможность запуска production-конфигурации с тестовым SECRET_KEY

В конфигурации проекта использовался fallback-ключ разработки. Если окружение было настроено неполно, приложение могло стартовать в production с небезопасным значением SECRET_KEY.

2.3. Утечка внутренних ошибок пользователю

При сбое отправки письма подтверждения регистрации пользователю возвращался текст исключения. Это могло раскрывать внутренние детали SMTP/API-инфраструктуры, сетевые адреса и техническую конфигурацию.

2.4. Отсутствие ограничения попыток на чувствительных маршрутах

Формы логина, регистрации и восстановления пароля были доступны без rate limiting. Это повышало риск brute force, enumeration и злоупотребления почтовыми сценариями.

2.5. Недостаточно жёсткие production-настройки Django

В проекте использовались базовые secure cookies, но отсутствовала часть важных защитных параметров: принудительный HTTPS-redirect, HSTS и дополнительные безопасные заголовки.

3. Реализованные меры защиты

3.1. Перевод destructive actions на POST

Для операций удаления были внедрены серверные ограничения: методы удаления теперь доступны только через POST, а формы в интерфейсе снабжены CSRF-токенами. 

core/views.py

@login_required
@require_POST
def delete_barber(request, barber_id):
    ...

@login_required
@require_POST
def delete_service(request, service_id):
    ...

@login_required
@require_POST
def delete_payment_method(request, method_id):
    ...

Интерфейс также был обновлен: вместо ссылок удаления теперь используются HTML-формы с POST. 

core/templates/settings_services.html
core/templates/settings_barbers.html
core/templates/settings_payment_methods.html

<form method="post" action="{% url 'delete_service' service.id %}" ...>
    {% csrf_token %}
    <button>Удалить</button>
</form>

3.2. Защита production от небезопасного SECRET_KEY

В конфигурации добавлена проверка: если приложение запускается не в DEBUG-режиме и SECRET_KEY не задан явно, система аварийно завершает запуск с ошибкой конфигурации. 

config/settings.py

SECRET_KEY = os.getenv("SECRET_KEY", "django-insecure-dev-only-key")

if not DEBUG and SECRET_KEY == "django-insecure-dev-only-key":
    raise ImproperlyConfigured("SECRET_KEY must be configured in production.")

3.3. Скрытие внутренних ошибок от пользователя

В логах ошибка по-прежнему фиксируется для разработчика, но на фронтенд выводится только безопасное и понятное сообщение без внутренних технических деталей. 

core/views.py

except Exception as exc:
    logger.exception("Failed to send activation email during registration")
    form.add_error(
        "email",
        "Не удалось отправить письмо подтверждения. Проверь настройки почты и попробуй ещё раз."
    )

3.4. Реализация базового rate limiting

Для защиты от brute force и abuse была реализована базовая система ограничения попыток по IP-адресу. Лимит действует на логин, регистрацию и восстановление пароля. После превышения допустимого числа попыток пользователь получает сообщение о временной блокировке. 

core/views.py

AUTH_RATE_LIMIT_WINDOW = 300
AUTH_RATE_LIMITS = {
    "login": 5,
    "register": 5,
    "password_reset": 5,
}

def get_client_ip(request):
    ...

def is_auth_rate_limited(scope, request):
    ...

class RateLimitedLoginView(auth_views.LoginView):
    ...

class RateLimitedPasswordResetView(auth_views.PasswordResetView):
    ...

Затем новые защищенные view были подключены в маршруты: 

config/urls.py

path("login/", RateLimitedLoginView.as_view(...), name="login")
path("password-reset/", RateLimitedPasswordResetView.as_view(...), name="password_reset")

3.5. Усиление production-настроек Django

В production-конфигурацию были добавлены дополнительные защитные параметры:

  • SESSION_COOKIE_SECURE
  • SESSION_COOKIE_HTTPONLY
  • CSRF_COOKIE_SECURE
  • CSRF_COOKIE_HTTPONLY
  • SECURE_SSL_REDIRECT
  • SECURE_HSTS_SECONDS
  • SECURE_HSTS_INCLUDE_SUBDOMAINS
  • SECURE_HSTS_PRELOAD
  • SECURE_CONTENT_TYPE_NOSNIFF
  • X_FRAME_OPTIONS = "DENY"
  • REFERRER_POLICY = "same-origin"
config/settings.py

if not DEBUG:
    STATICFILES_STORAGE = "whitenoise.storage.CompressedManifestStaticFilesStorage"
    SESSION_COOKIE_SECURE = True
    SESSION_COOKIE_HTTPONLY = True
    CSRF_COOKIE_SECURE = True
    CSRF_COOKIE_HTTPONLY = True
    SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")
    SECURE_SSL_REDIRECT = os.getenv("SECURE_SSL_REDIRECT", "True").lower() == "true"
    SECURE_HSTS_SECONDS = int(os.getenv("SECURE_HSTS_SECONDS", "31536000"))
    SECURE_HSTS_INCLUDE_SUBDOMAINS = True
    SECURE_HSTS_PRELOAD = True
    SECURE_CONTENT_TYPE_NOSNIFF = True
    X_FRAME_OPTIONS = "DENY"
    REFERRER_POLICY = "same-origin"

5. Итоговый результат

В результате проведенной работы безопасность проекта KZCRMS была заметно усилена. Были устранены наиболее критичные риски, связанные с удалением данных, brute force, небезопасной production-конфигурацией и информационными утечками через интерфейс. Реализованные изменения повышают устойчивость системы к распространенным веб-угрозам и делают проект более подготовленным к использованию реальными клиентами.

7. Заключение

Выполненный аудит и доработка безопасности показали, что даже небольшой SaaS-проект требует системной проработки защитных механизмов. На практике были применены принципы безопасной конфигурации, защиты от CSRF, ограничения атак перебора, скрытия внутренних ошибок и автоматизированной проверки внесённых изменений.
